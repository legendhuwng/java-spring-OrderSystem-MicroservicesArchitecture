## 🥇 PHASE 0 – Chuẩn bị (KHÔNG VIẾT CODE NGAY)

⏱ 2–3 ngày

### Việc cần làm

* Viết **scope** (chỉ làm những gì cần)
* Chọn:
  * Saga **Orchestrator** (Order Service)
  * Database **per service**
* Vẽ **1 sơ đồ duy nhất**:
  * Services
  * Kafka topics
  * Flow order

### ⚠️ Định nghĩa Compensating Transaction NGAY TỪ ĐÂY

Đây là phần hay bị bỏ qua và gây fail về sau. Cần xác định rõ:

| Bước Saga          | Event thành công      | Compensating khi thất bại                          |
| ------------------ | --------------------- | -------------------------------------------------- |
| Order Service      | order-created         | order-cancelled                                    |
| Inventory Service  | inventory-reserved    | inventory-released (khi payment fail)              |
| Payment Service    | payment-success       | payment-refunded (khi shipping fail)               |
| Shipping Service   | shipping-created      | (điểm cuối – không rollback)                       |

👉 Nếu bỏ qua phase này → 90% fail khi implement Saga rollback.

## 🥈 PHASE 1 – Core Infrastructure (BẮT BUỘC TRƯỚC)

⏱ 3–5 ngày

### 1️⃣ Tạo Project Skeleton

* Mono-repo (dễ cho 1 người)
* Spring Boot 3 + Java 17

### 2️⃣ Docker Compose (BỔ SUNG MỚI – làm ngay)

> ⚠️ Kafka + Zookeeper + Oracle chạy tay rất đau. Bắt buộc có docker-compose.yml từ Phase 1.

docker-compose.yml bao gồm:
* Zookeeper + Kafka
* Oracle XE (hoặc PostgreSQL nếu không bắt buộc Oracle)
* Kafka UI (Kowl / Redpanda Console) – giúp debug message dễ hơn nhiều

### 3️⃣ Các service nền

**LÀM TRƯỚC – chưa business**

| Thứ tự | Service                    | Ghi chú                                  |
| ------ | -------------------------- | ---------------------------------------- |
| 1      | Config Server (optional)   | Có thể bỏ nếu scope nhỏ                  |
| 2      | Service Registry (Eureka)  | ✅ Đã có – port 8761                      |
| 3      | API Gateway                | ✅ Đã có – port 8080, có CorrelationIdFilter |
| 4      | Kafka + Zookeeper          | Chạy qua Docker Compose                  |
| 5      | Oracle DB / PostgreSQL     | Chạy qua Docker Compose                  |

👉 Chỉ cần **chạy được**:
* Gateway → Registry → 1 service dummy
* Kafka send/consume test message
* docker-compose up -d chạy toàn bộ infrastructure

## 🥉 PHASE 2 – ORDER SERVICE (LÀM TRƯỚC TẤT CẢ)

⏱ 5–7 ngày

> ❗ **Order Service = trái tim hệ thống**
> Là Saga Orchestrator – giao tiếp với mọi service – quyết định trạng thái toàn hệ thống.

### REST API

* `POST /orders`
* `GET /orders/{id}`

### Database

* Bảng ORDER: id, status, total_amount, created_at, updated_at
* Bảng ORDER_ITEM: id, order_id, product_id, quantity, price
* Bảng OUTBOX: id, aggregate_type, aggregate_id, event_type, payload, status, created_at  ← BỔ SUNG MỚI

### ⭐ Transactional Outbox Pattern (BỔ SUNG MỚI)

> ⚠️ Vấn đề: Lưu DB thành công nhưng publish Kafka thất bại → order mất.
> Outbox pattern giải quyết điều này.

Cách hoạt động:
* Khi tạo order: lưu ORDER và OUTBOX trong **cùng 1 transaction**
* Một scheduler/thread riêng đọc OUTBOX và publish lên Kafka
* Sau khi publish thành công: update status = PUBLISHED

💡 Interviewer rất ấn tượng với pattern này. Nên implement ngay từ Order Service.

### Kafka

* Publish: `order-created`
* Listen: `inventory-reserved`, `inventory-failed`, `payment-success`, `payment-failed`, `shipping-created`

### Order Status Flow

```
PENDING → INVENTORY_CHECKING → PAYMENT_PROCESSING → SHIPPING → COMPLETED
PENDING → CANCELLED  (khi inventory hoặc payment fail)
```

👉 Test:
**Tạo order → lưu DB → publish Kafka OK (kể cả khi Kafka tạm thời down)**

## 🏅 PHASE 3 – INVENTORY SERVICE

⏱ 4–5 ngày

### Làm gì?

* Consumer `order-created`
* Reserve stock
* Oracle inventory
* Publish:
  * `inventory-reserved`
  * `inventory-failed`

### ⚠️ Idempotency – LÀM NGAY TẠI ĐÂY (BỔ SUNG MỚI)

> Kafka có thể deliver cùng 1 message nhiều lần (at-least-once delivery).
> Nếu không idempotent → reserve stock 2 lần cho cùng 1 order.

Cách xử lý:
* Lưu event_id đã xử lý vào bảng PROCESSED_EVENTS
* Trước khi xử lý: check xem event_id đã tồn tại chưa
* Nếu đã có → skip, không xử lý lại
* Áp dụng pattern này cho **TẤT CẢ consumer** từ đây về sau

### Kết quả mong muốn

```
Create Order → Inventory reserve → Order updated status
```

👉 Đến đây bạn đã có **mini Saga** chạy được.

---

## 🏆 PHASE 4 – PAYMENT SERVICE

⏱ 4–6 ngày

### Làm gì?

* Consumer `inventory-reserved`
* Fake payment (mock – không cần tích hợp thật)
* Publish:
  * `payment-success`
  * `payment-failed`

### Order Service lắng nghe:

* `payment-success` → update status → PAYMENT_PROCESSING → SHIPPING
* `payment-failed` → publish `inventory-release-command` → Inventory release stock → Order CANCELLED

### Saga Rollback khi payment-failed

```
payment-failed
  → Order Service publish inventory-release-command
  → Inventory Service release stock
  → Order status → CANCELLED
```

👉 Tới đây bạn đã có:

```
Order → Inventory → Payment  (kể cả rollback)
```
